## Почему `Юнит-экономика…csv` не совпадает с «Начислено за период»

Файл `Юнит-экономика_…csv` даёт другие итоги (в этом периоде: “net” около **1 453 ₽**), потому что:

- это **другой отчёт** с другой моделью отнесения по периоду/по событиям;
- там другие колонки (часть начислений может считаться иначе, и/или попадать в другие разделы).

Вывод: для мэтчинга главной карточки «Начислено за период» нужно брать эталон из:
- `Отчет по начислениям…csv` и/или
- `Отчет по товарам…csv`

а `Юнит-экономика…csv` — отдельная задача (юнит-экономика), не критерий истины для карточки начислений.

## Где у нас в проекте вероятнее всего возникает расхождение (по коду)

Ниже — причины, которые **с высокой вероятностью** дают несовпадение именно по комиссии/агентам/логистике.

### 1) Неправильная дата для фильтра периода (ключевая причина)

В ЛК период задаётся по **дате начисления** (`Дата начисления` в CSV).

В текущей логике синка/деталей дата берётся как:

- `date = operation_date[:10]`

Если `operation_date` ≠ `Дата начисления`, то выборка за период будет другая → суммы не совпадут.

**Требование для мэтчинга:** в нашей базе и API должна быть возможность резать начисления **так же, как ЛК**:
- либо хранить `accrual_date` отдельно и фильтровать по нему,
- либо доказать, что используемая дата эквивалентна ЛК (на практике часто нет).

### 2) Использование `abs()` ломает сторно/корректировки (особенно “эквайринг”)

В `Отчет по товарам…csv` встречается сторно по эквайрингу:
- строка **−7.08 ₽** и отдельная строка **+7.08 ₽** (в сумме 0)

Если при агрегации применять `abs()`, то обе строки станут по модулю положительными и “расходы” завысятся.

**Требование:** в каноническом слое начислений (ledger) хранить суммы **со знаком**, без `abs()`.
`abs()` допустим только на уровне UI (форматирование) и только там, где это нужно для отображения.

### 3) Проценты в дереве считаются от `mp_sales.revenue`, а ЛК — от “Продажи”

В ЛК “Продажи 7 097 ₽” включают:
- Выручку (денежную) + Баллы + Партнёров

Если в `%` использовать только `mp_sales.revenue` (обычно это “денежная выручка”), то:
- `%` комиссии/логистики в UI будет отличаться от ЛК,
- даже если суммы в ₽ совпадают.

**Требование:** базу для процентов нужно синхронизировать с ЛК:
- либо считать `%` от “Продажи” в дереве (Выручка+Баллы+Партнёры),
- либо явно подписать в UI “% от выручки (без баллов/партнёров)” — но тогда это уже НЕ 1‑в‑1.

### 4) Не хватает компонент “Продажи”: баллы и партнёры

Если наша модель строится из Finance операций, то нужно обеспечить, что в “Продажи” попадают три компоненты:
- Выручка
- Баллы за скидки
- Программы партнёров

Если добавлена только “Выручка”, то “Продажи” будут ниже эталона на **2 226.50 ₽** в этом периоде → дальше всё дерево/проценты “поплывёт”.

## Каноническая модель для мэтчинга (рекомендуемая)

Чтобы стабильно совпадать с ЛК, нужен «канонический слой начислений» (ledger), где для каждой строки есть:

- **accrual_date**: дата начисления (как в ЛК)
- **event_date**: дата события (если нужна аналитика по событиям)
- **type / group**: как в ЛК (Группа услуг / Тип начисления)
- **amount_signed**: сумма со знаком
- **идентификаторы**: operation_id / posting_id / отправление / sku / product_id

А уже из ledger строится агрегат:
- «карточка» (итоги)
- дерево затрат
- проценты

## Реализовано (фиксы мэтчинга) — 23.01.2026

### Что изменили в проекте

- **Маппинг “LastMile”**:
  - `LastMile` → `Услуги агентов / Доставка до места выдачи`
  - `Logistic` → `Услуги доставки / Логистика`
- **Убрали мультипликацию сумм по items**:
  - order‑level суммы распределяются по `items[]` пропорционально `quantity` (чтобы эквайринг/логистика/комиссия не умножались при нескольких товарах в одной операции)
- **Ровные копейки для операций без items** (например, складские услуги):
  - распределение делается в **копейках** с остатком, чтобы сумма распределения всегда была равна исходной сумме до 0.01 ₽
- **Проценты как в ЛК**:
  - `%` считается от **“Продажи”** (категория дерева), а не от `mp_sales.revenue`
  - в ответ добавлено поле `percent_base_sales`
