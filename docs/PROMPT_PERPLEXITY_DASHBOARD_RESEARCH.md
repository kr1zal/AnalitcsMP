# Промпт для Perplexity: исследование компоновки главной страницы дашборда

Скопируйте блок ниже в Perplexity для детального исследования по подбору и компоновке информации, графиков и дашбордов.

## Блок для копирования

```
Ты — эксперт по проектированию аналитических дашбордов и data-driven интерфейсов с опытом 10+ лет в продуктах класса B2B SaaS, e-commerce analytics и BI. Ты владеешь: information architecture (IA), визуальной иерархией, выбором типов графиков под задачу (Chart selection), best practices Stripe / Shopify / GA4 / Amplitude / Mixpanel, принципами perceptual cognition (как пользователь сканирует экран, F- и Z-паттерны), responsive-компоновкой для desktop и mobile, accessibility (WCAG) для данных и графиков. Ты опираешься на исследования Nielsen Norman Group, Stephen Few (Show Me the Numbers, Now You See It), Edward Tufte, а также на реальные кейсы современных дашбордов (Stripe Dashboard, Shopify Analytics, Google Analytics 4, Metabase, Retool).

Контекст продукта:
- Интерактивный дашборд аналитики для продавцов на маркетплейсах Wildberries и Ozon.
- Пользователь: владелец/менеджер малого бизнеса, 5–50 SKU, нужны быстрые ответы: «Как идут продажи?», «Сколько ушло на удержания МП?», «Какая прибыль?», «Какой ДРР?».
- Данные: выручка, прибыль (оценка), ДРР, рекламные расходы, расходы МП (комиссия, логистика, хранение и т.д.), к перечислению, сравнение с предыдущим периодом; разбивка по WB и Ozon; графики по дням (заказы, выкупы, выручка, средний чек, ДРР); таблица остатков по складам; карточки начислений с детализацией удержаний.
- Ограничения: главная страница одна (overview), без перегрузки; mobile-first, но desktop тоже важен; стек: React, Recharts, Tailwind.

Задача — провести исследование и дать структурированные, пригодные к внедрению рекомендации по следующим пунктам. Отвечай по-русски, с обоснованием и ссылками на практики/источники где уместно.

1) Информационная архитектура главной страницы (Overview)
- В каком порядке логичнее располагать блоки: сводные KPI (карточки) → разбивка по каналам (WB/Ozon) → графики по времени → таблицы (остатки и т.д.)? Или есть доказательно лучшие альтернативы для e-commerce/маркетплейс-аналитики?
- Сколько метрик «above the fold» на desktop и на mobile считается оптимальным без когнитивной перегрузки? Какие метрики приоритизировать первыми (выручка, прибыль, ДРР, реклама, что ещё)?
- Нужен ли на главной единый «главный номер» (hero metric) и если да — какой и как его визуально выделить?

2) Карточки метрик (Summary cards / KPI cards)
- Рекомендуемое количество карточек в одном ряду на desktop (4, 6, 8?) и на mobile (2, 4 в два ряда, scroll?). Размер шрифта для основного числа и для подписи по гайдам (например Apple HIG, Material, или Stripe-like).
- Нужны ли на каждой карточке: тренд (↑/↓ и %), сравнение с прошлым периодом, tooltip с расшифровкой? Как не перегрузить карточку при ограниченном месте на mobile?
- Иконки на карточках: всегда, только на desktop, или только для части метрик? Рекомендации по семантике (выручка — одна иконка, расходы — другая).

3) Выбор типов графиков (Chart selection)
- Для временного ряда «продажи/выручка по дням»: Line, Area, Bar — когда что предпочтительнее? Учитывая сравнение заказов/выкупов/выручки в одном блоке (табы или несколько линий).
- Для «средний чек по дням»: Bar chart vs Line — что лучше по читаемости и по исследованиям?
- Для ДРР (доля рекламных расходов в выручке): один слой Area, Line, или комбинация с baseline 0? Нужна ли опорная линия (например целевой ДРР)?
- Сколько разных графиков на одной странице не перегружают пользователя? Рекомендации по высоте графика (px или vh) для desktop и mobile.

4) Компоновка и визуальная иерархия
- Grid: фиксированное количество колонок (12, 8) vs fluid; рекомендуемая ширина контейнера (max-width) для контента дашборда.
- Отступы между блоками (карточки, графики, таблицы): по системе 8px, или 4/8/16/24 — какие конкретные значения для gap между секциями?
- Как визуально отделить «сводку» от «детализации» (рамки, фон, отступы, заголовки секций)? Нужны ли явные заголовки секций (например «Ключевые метрики», «Динамика продаж», «Остатки») и в каком стиле (размер, вес, цвет)?

5) Таблицы на главной
- Остатки по складам: показывать на главной полную таблицу, свёрнутую (top N + «ещё»), или только суммарные цифры с переходом в отдельный экран? Критерии выбора.
- Рекомендуемое количество видимых строк без скролла на desktop и на mobile. Сортировка по умолчанию (критичные остатки вверху — да/нет и почему).

6) Фильтры и период
- Где размещать фильтры (период 7/30/90д, кастомные даты, маркетплейс, товары): только сверху, слева как сайдбар, или комбинированно? Что чаще встречается в успешных дашбордах и почему?
- На mobile: фильтры в одну строку с горизонтальным скроллом, выпадающие панели, или всегда видимый компактный блок — плюсы и минусы.

7) Сравнение с предыдущим периодом и тренды
- Как показывать «выручка за период» vs «выручка за предыдущий период» и дельту: отдельные карточки, одна карточка с подписью (↑ +12%), или отдельная мини-визуализация (sparkline)?
- Рекомендации по цвету для положительной/отрицательной дельты и по доступности (цвет + не только цвет).

8) Специфика mobile
- Приоритизация контента на главной для mobile: что оставить above the fold в первую очередь?
- Графики на mobile: уменьшенная высота vs горизонтальный скролл vs табы «график 1 / график 2» — что предпочтительнее по исследованиям и практике?
- Таблицы: карточный вид (каждая строка — карточка) vs горизонтальный скролл таблицы — когда что применять.

9) Референсы и антипаттерны
- Назови 3–5 конкретных дашбордов (продуктов) с сильной компоновкой главной страницы и что у них взять.
- Перечисли частые антипаттерны: перегруженность главной, неподходящий тип графика, нарушение иерархии, плохая читаемость на mobile.

10) Чек-лист для ревью главной страницы
- Дай краткий чек-лист (10–15 пунктов), по которому можно проверить текущую компоновку дашборда перед релизом (от «есть ли один явный фокус» до «есть ли fallback при пустых данных»).

Ответ оформи структурированно по пунктам 1–10. Где возможно — со ссылками на исследования, гайды (NN/g, WCAG, vendor design systems) или примеры продуктов. Если по какому-то пункту нет однозначной нормы — опиши варианты и критерии выбора.
```

## Скрины текущего состояния

**Да, стоит сделать скрины того, что сейчас есть** — так исследование в Perplexity (или ревью дизайнера) будет опираться на реальный интерфейс, а не только на описание.

### Что снять

| Что | Зачем |
|-----|--------|
| **Desktop — вся главная** (от фильтров до конца таблицы) | Общая компоновка, порядок блоков, плотность |
| **Desktop — первый экран (above the fold)** | Что видно без скролла |
| **Mobile — вся главная** | Компоновка на узком экране |
| **Mobile — первый экран** | Приоритет контента на телефоне |
| По желанию: **крупно блок карточек**, **один из графиков**, **таблица остатков** | Детали по типам компонентов |

### Куда сохранять

- Папка `docs/screenshots/` (создай, если нет) или `frontend/public/screenshots/`.
- Имена файлов с датой и контекстом, например:  
  `dashboard-main-desktop-2026-01-30.png`,  
  `dashboard-main-mobile-2026-01-30.png`,  
  `dashboard-cards-desktop-2026-01-30.png`.

### Как использовать с Perplexity

1. Сначала отправь текстовый промпт из блока выше.
2. В следующем сообщении приложи скрины и напиши, например:  
   *«Вот как сейчас выглядит главная (desktop и mobile). Дай рекомендации по пунктам 1, 4 и 8 с опорой на эти экраны.»*  
   Perplexity поддерживает загрузку изображений — по ним можно получить конкретные советы («карточки слишком плотные», «график лучше заменить на Area» и т.д.).

### Перед съёмкой

- Запусти frontend (`npm run dev`), выбери период с данными (например 7д или 30д).
- На desktop открой на полную ширину окно (или 1280–1440px).
- На mobile: DevTools → Toggle device toolbar, выбери iPhone или 390×844.
