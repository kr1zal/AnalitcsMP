## 1. Источники данных

Система использует пять ключевых источников. Понимание различий между ними критично для корректности расчётов.

### 1.1. `mp_sales.revenue` — аналитика маркетплейсов

Все заказы, включая непроведённые (ещё не отражённые в финансовом отчёте). Это данные из API аналитики МП: WB `get_orders` + `get_sales`, Ozon `get_posting_fbs_list` + `get_posting_fbo_list`.

- Содержит: `orders_count`, `sales_count`, `returns_count`, `revenue`
- Группировка: по дням, по товарам, по маркетплейсам
- Особенность: `revenue` здесь может включать заказы, которые ещё не прошли через финансовый отчёт (не проведены)

**Когда используется:** как база для расчёта долей (`costsTreeRatio`), для графика продаж, для плана продаж (actual revenue), для воронки заказов.

### 1.2. `costs-tree "Продажи"` — финализированные продажи

Данные из финансового отчёта МП (`mp_costs_details`). Извлекаются через SQL RPC `get_costs_tree()`. Категория "Продажи" содержит только проведённые (settled) суммы.

- WB источник: `reportDetailByPeriod` (финотчёт)
- Ozon источник: `get_finance_transaction_list` (Finance API v3)
- Агрегация: `SUM(amount WHERE category = 'Продажи')` — всегда положительное число

**Когда используется:** для расчёта `costsTreeRatio`, для отображения "Выкупов" на дашборде, для карточек МП.

### 1.3. `costs-tree total_accrued` — к перечислению (payout)

Итоговая сумма, которую маркетплейс начислил продавцу за период. Рассчитывается как `SUM(ALL amounts)` по всем категориям дерева удержаний.

```
total_accrued = Продажи + Credits + Удержания
             = Σ(все tree items)
```

- Включает: положительные (продажи, СПП, возмещения) и отрицательные (комиссия, доставка, хранение)
- Это реальная сумма денег, которую продавец получит от МП

**Когда используется:** как отправная точка для расчёта прибыли (вместо revenue).

### 1.4. `mp_ad_costs` — рекламные расходы

Данные по рекламным кампаниям: WB Продвижение и Ozon Performance. Хранятся по дням, по кампаниям, с привязкой к товарам.

- Поля: `cost`, `impressions`, `clicks`, `orders_count`, `campaign_id`, `product_id`
- Если `product_id = NULL` — неатрибутированная реклама, распределяется пропорционально выручке

### 1.5. `mp_products.purchase_price` — себестоимость

Закупочная цена за единицу товара. Задаётся пользователем вручную в разделе "Настройки -> Товары". API маркетплейсов (в частности Ozon) не предоставляют данные о себестоимости.

- Если `purchase_price = 0` — на дашборде отображается предупреждение
- Закупка в формулах: `purchase = purchase_price * sales_count`

## 2. Dashboard Profit (главная формула)

Ключевая задача системы — рассчитать точную чистую прибыль продавца.

### Формула

```
profit = total_payout - purchase_adjusted - ads_adjusted
```

Где:

| Компонент | Формула | Описание |
|-----------|---------|----------|
| `total_payout` | `costs_tree.total_accrued` | К перечислению от МП (после всех удержаний) |
| `purchase_adjusted` | `purchase * costsTreeRatio` | Себестоимость, скорректированная на долю проведённых |
| `ads_adjusted` | `ad_cost` (без коррекции) | Рекламные расходы за период |

### costsTreeRatio — зачем нужен коэффициент

```
costsTreeRatio = costs_tree_SALES / mp_sales_revenue
```

**Проблема.** Не все заказы за период проведены маркетплейсом. Например, из 100 000 руб. заказов финотчёт содержит только 80 000 руб. Если взять себестоимость на все 100 000 — прибыль будет занижена.

**Решение.** Коэффициент `costsTreeRatio` показывает, какая доля заказов (по сумме) уже проведена через финансовый отчёт. Себестоимость умножается на этот коэффициент, чтобы учитывать только реально проведённые товары.

**Пример:**
```
mp_sales_revenue (все заказы)    = 100 000 руб.
costs_tree_SALES (проведённые)   =  80 000 руб.
costsTreeRatio                   = 80 000 / 100 000 = 0.80

purchase (полная)                = 30 000 руб.
purchase_adjusted                = 30 000 * 0.80 = 24 000 руб.

total_payout                     = 56 000 руб.
ads                              =  5 000 руб.

profit = 56 000 - 24 000 - 5 000 = 27 000 руб.
```

### Граничные случаи

| Ситуация | costsTreeRatio | Поведение |
|----------|----------------|-----------|
| Все заказы проведены | 1.0 | Себестоимость не корректируется |
| costs_tree_SALES = 0 | 1.0 (fallback) | Нет данных финотчёта — используем полную себестоимость |
| costs_tree_SALES > mp_sales | 1.0 (fallback) | Аномалия данных — безопасный fallback |
| payout недоступен | fallback на `summary.net_profit` | Backend-расчёт через RPC |

### Реализация

**Backend** (`dashboard.py`, строки 148-192):
```python
costs_tree_ratio = (
    costs_tree_sales / total_mp_sales_revenue
    if total_mp_sales_revenue > 0 and 0 < costs_tree_sales < total_mp_sales_revenue
    else 1.0
)
```

**Frontend** (`DashboardPage.tsx`, строки 428-441):
```typescript
const costsTreeRatio =
    summaryRevenue > 0 && pureSalesForRatio > 0 && pureSalesForRatio < summaryRevenue
      ? pureSalesForRatio / summaryRevenue
      : 1;
```

### Важно: credits НЕ входят в ratio

`costsTreeRatio` использует только чистые "Продажи" (`costs_tree_SALES`), без credits (СПП, возмещения). Credits — это компенсация от МП, а не доход от продаж. Если включить credits в числитель, коэффициент будет завышен, а себестоимость — недостаточно скорректирована.

## 3. СПП и Credits (WB)

### Что такое СПП

**СПП (Скидка Постоянного Покупателя)** — механизм Wildberries, при котором часть цены товара оплачивается не покупателем, а самим маркетплейсом. Покупатель видит сниженную цену, разница компенсируется продавцу отдельной строкой в финотчёте.

Пример: товар стоит 1 000 руб., СПП = 200 руб. Покупатель платит 800 руб., WB доплачивает 200 руб. продавцу.

### Формула отображаемой выручки

```
displayed_revenue = costs_tree_sales + credits
```

Где:
- `costs_tree_sales` — tree item "Продажи" (чистые продажи из финотчёта)
- `credits` — все положительные tree items, кроме "Продажи" (СПП, возмещения)

```
credits = tree items WHERE amount > 0 AND name != "Продажи"
```

### Почему credits ВХОДЯТ в displayed_revenue

Покупатель получает товар, продавец получает полную сумму (включая СПП). Для продавца это реальный доход. В Личном кабинете WB "Продажи" тоже включают СПП. Система показывает данные так, как их видит продавец.

### Почему credits НЕ ВХОДЯТ в costsTreeRatio

```
costsTreeRatio = costs_tree_SALES / mp_sales_revenue   (БЕЗ credits!)
```

`costsTreeRatio` отражает долю проведённых заказов. Credits — это не заказы, а компенсация от маркетплейса. Если включить их в числитель, коэффициент покажет неверную картину проведённости.

### Почему двойного учёта СПП нет

Распространённая ошибка — предположение, что СПП может быть учтена дважды: в продажах и в credits. Это не так:

1. `total_accrued` (payout) = `SUM(ALL amounts)` — содержит СПП как одну из строк
2. `displayed_revenue` используется ТОЛЬКО для отображения на карточках
3. Прибыль считается от `total_accrued` (payout), а НЕ от `displayed_revenue`
4. `costsTreeRatio` использует чистые sales (без credits)

**Инвариант** (проверяется на реальных данных):
```
displayed_revenue + удержания = начислено (total_accrued)
```

Пример:
```
Продажи (pure)  = 7 866 руб.
Credits (СПП)   = 3 243 руб.
Удержания       = -2 713 руб.
────────────────────────────────
total_accrued   = 7 866 + 3 243 + (-2 713) = 8 396 руб.
displayed_rev   = 7 866 + 3 243             = 11 109 руб.
удержания (abs) = 2 713 руб.
displayed_rev - удержания = 11 109 - 2 713  = 8 396 = total_accrued  (сходимость)
```

### Отличие от OZON

У OZON нет credits (нет СПП или аналога). Все положительные суммы — только в категории "Продажи". Поэтому для OZON: `displayed_revenue = costs_tree_sales` (credits = 0).

## 4. UE Profit (per product)

Юнит-экономика (Unit Economics) рассчитывает прибыль по каждому товару отдельно.

### Формула

```
profit_i = total_payout * (revenue_i / sum_revenue) - purchase_i * ratio - ad_i * ratio
```

Где:

| Компонент | Описание |
|-----------|----------|
| `total_payout` | Общий payout (из costs-tree, сумма по всем МП) |
| `revenue_i` | Выручка товара i из `mp_sales` (аналитика) |
| `sum_revenue` | Общая выручка всех товаров из `mp_sales` |
| `purchase_i` | `purchase_price_i * sales_count_i` (себестоимость товара) |
| `ratio` | `costsTreeRatio` (коэффициент проведённости) |
| `ad_i` | Рекламные расходы, привязанные к товару i |

### Гарантия сходимости

```
sum(profit_i для всех i) = Dashboard_profit
```

Это обеспечивается пропорциональным распределением `total_payout` по товарам:
- Каждый товар получает долю payout, равную его доле в общей выручке (`revenue_i / sum_revenue`)
- Себестоимость и реклама вычитаются из этой доли
- Поскольку `sum(revenue_i / sum_revenue) = 1`, сумма UE-прибылей строго равна Dashboard-прибыли

### Альтернативный подход (и почему он не используется)

Наивный подход `profit_i = revenue_i - costs_i - purchase_i - ad_i` не работает, потому что:
- `revenue_i` из `mp_sales` включает непроведённые заказы
- `costs_i` из `mp_costs` может не соответствовать `revenue_i` по периоду
- Суммы UE не сойдутся с Dashboard

### Отображаемые метрики UE

Для каждого товара в таблице:

| Метрика | Формула |
|---------|---------|
| Выручка | `costs_tree_revenue * (revenue_i / sum_revenue)` |
| Удержания МП | `displayed_revenue_i - payout_share_i` |
| Себестоимость | `purchase_price * sales_count * costsTreeRatio` |
| Реклама | `ad_cost_i` (из `mp_ad_costs`) |
| Прибыль | `payout_share_i - purchase_adj_i - ad_i` |
| Прибыль/шт | `profit_i / sales_count_i` |
| ДРР | `ad_cost_i / displayed_revenue_i * 100%` |

### Неатрибутированная реклама

Если у рекламной записи `product_id = NULL` (кампания не привязана к товару), расходы распределяются пропорционально выручке:

```python
for pid, m in product_metrics.items():
    share = m["revenue"] / total_mp_sales_revenue
    ad_by_product[pid] += unattributed_ad * share
```

## 5. Per-Marketplace Profit

Прибыль рассчитывается отдельно для каждого маркетплейса (OZON / WB) на карточках MarketplaceBreakdown.

### Формула

```
profit_mp = payout_mp - purchase * share - ad * share
share     = pureSales_mp / totalPureSales
```

Где:
- `payout_mp` — `total_accrued` из costs-tree конкретного МП
- `pureSales_mp` — чистые "Продажи" этого МП (tree item "Продажи", без credits)
- `totalPureSales` — сумма чистых продаж по всем МП (`ozonPureSales + wbPureSales`)
- `purchase` — скорректированная себестоимость (`adjustedPurchase`)
- `ad` — общие рекламные расходы (`summary.ad_cost`)

### Гарантия сходимости

```
profit_ozon + profit_wb = dashboard_profit
```

Это работает потому что:
```
(payout_ozon + payout_wb) - purchase * (share_ozon + share_wb) - ad * (share_ozon + share_wb)
= total_payout - purchase * 1 - ad * 1
= dashboard_profit
```

Где `share_ozon + share_wb = 1` по определению.

### Проверено на реальных данных

```
profit_ozon = 1 013.40 руб.
profit_wb   = 0.00 руб.    (период без WB данных)
dashboard   = 1 013.40 руб.
Сходимость: 1 013.40 = 1 013.40
```

## 6. DRR (Доля Рекламных Расходов)

### Формула

```
drr = ad_cost / revenue * 100%
```

Где:
- `ad_cost` — сумма всех рекламных расходов за период (из `mp_ad_costs`)
- `revenue` — выручка за период (зависит от контекста: карточка dashboard или UE-таблица)

### Применение в разных контекстах

| Контекст | `revenue` | Описание |
|----------|-----------|----------|
| Dashboard карточка | `revenueForTile` (из costs-tree) | Общий ДРР по всем товарам |
| UE-таблица (per product) | `displayed_revenue_i` | ДРР конкретного товара |
| Ads page (по дням) | `mp_sales.revenue` по дню | Дневной ДРР для графика |
| Рекламная кампания | `total_revenue` (все продажи) | ДРР кампании от общей выручки |

### Интерпретация

| ДРР | Оценка |
|-----|--------|
| 0-5% | Низкий (эффективная реклама) |
| 5-15% | Средний (норма для большинства категорий) |
| 15-30% | Высокий (требует анализа ROI) |
| >30% | Критический (реклама может быть убыточной) |

## 7. Stock Forecast (Прогноз остатков)

### Формула

```
days_remaining = total_quantity / avg_daily_sales(30d)
avg_daily_sales = SUM(sales_count за 30 дней) / 30
```

Где:
- `total_quantity` — текущий остаток товара на всех складах (из `mp_stocks`)
- `avg_daily_sales` — средние дневные продажи за последние 30 дней (из `mp_sales.sales_count`)

### Цветовые пороги

| Цвет | Условие | Статус |
|------|---------|--------|
| Красный | `days_remaining <= 7` | Критический — товар закончится через неделю |
| Янтарный | `7 < days_remaining <= 14` | Предупреждение — требуется заказ поставки |
| Синий | `14 < days_remaining <= 30` | Нормальный — запас на месяц |
| Зелёный | `days_remaining > 30` | Достаточный запас |

### Граничные случаи

| Ситуация | `days_remaining` |
|----------|------------------|
| Нет продаж за 30 дней | `null` (не отображается) |
| Товара нет на складе | `null` |
| `avg_daily_sales = 0` | `null` |
| Товар только на одном МП | Считается по данному МП |

### Статусы складских остатков

| Статус | Условие | Визуал |
|--------|---------|--------|
| OOS (Out of Stock) | `quantity = 0` | Красный бэдж |
| Критический | `quantity < 20` | Красный текст |
| Низкий | `quantity < 100` | Янтарный текст |
| OK | `quantity >= 100` | Зелёный текст |

## 8. Sales Plan Completion (План продаж)

### Три уровня плана

Система поддерживает три уровня планирования с чётким приоритетом:

```
1. Total plan (общий)      →  mp_sales_plan_summary, level="total"
2. Per-MP plan (по МП)     →  mp_sales_plan_summary, level="wb"/"ozon"
3. Per-product plan        →  mp_sales_plan (по товарам, с marketplace)
```

### Приоритет при расчёте completion

```
if total_plan > 0:
    use total_plan                    # Приоритет 1
elif sum(mp_plans) > 0:
    use sum(mp_plans for active_mps)  # Приоритет 2
elif sum(product_plans) > 0:
    use sum(product_plans)            # Приоритет 3
else:
    plan_level = "none"               # Нет плана
```

### Формула completion

```
completion_percent = (actual_revenue / plan_revenue) * 100%
```

### Критический багфикс: actual только за месяцы с планом

**Проблема** (до исправления): если пользователь выбрал диапазон "январь-февраль", а план задан только на февраль, actual считался за оба месяца. Это завышало completion.

**Решение** (`_plan_months_range`): actual revenue считается ТОЛЬКО за те месяцы, в которых есть план.

```python
plan_months = sorted(months_with_summary.get("total", set()))
actual_from, actual_to = _plan_months_range(plan_months, today)
```

### Критический багфикс: active_mps

**Проблема**: при `marketplace=all` и плане только на WB, actual суммировал WB + Ozon. Если Ozon приносил больше, completion показывал >100%.

**Решение**: список `active_mps` — actual считается только по тем МП, на которые есть план.

```python
active_mps = []
if summary_by_level.get("wb", 0) > 0:
    active_mps.append("wb")
if summary_by_level.get("ozon", 0) > 0:
    active_mps.append("ozon")
```

### Footer = взвешенное completion

В UE-таблице footer-строка показывает общий completion. Используется взвешенное среднее (от backend `completion_percent`), а НЕ простое среднее `sum(percents) / N`.

### Pace и Forecast (v2)

```
pace_daily      = actual / days_elapsed
required_pace   = (plan - actual) / days_remaining
forecast_revenue = actual + pace_daily * days_remaining
forecast_percent = (forecast_revenue / plan) * 100%
```

Где:
- `days_elapsed` — дней прошло с начала месяца (или периода планирования)
- `days_remaining` — дней осталось до конца месяца

## 9. Conversion (Конверсия выкупа)

### Формула

```
conversion = (sales / orders) * 100%
```

Где:
- `sales` — количество выкупленных товаров (`sales_count` из `mp_sales`)
- `orders` — количество заказов (`orders_count` из `mp_sales`)

### Контекст

| Метрика | Описание |
|---------|----------|
| Заказ (`order`) | Покупатель оформил заказ на МП |
| Выкуп (`sale`) | Покупатель забрал и оплатил товар |
| Возврат (`return`) | Покупатель вернул товар |

Конверсия выкупа показывает, какой процент заказанных товаров покупатели фактически забрали. Для WB типичная конверсия — 40-70% (покупатели заказывают несколько размеров и возвращают лишнее). Для Ozon конверсия обычно выше.

### Отображение

- График ConversionChart: AreaChart по дням, голубой цвет (#0ea5e9)
- Тултип: конверсия %, количество заказов и выкупов
- Lazy-loaded (Recharts)

## 10. WB Методология

**Статус: ВЕРИФИЦИРОВАНА на реальных данных. Аудит 15.02.2026.**

### Источник данных

Таблица `mp_costs_details` — данные из WB финансового отчёта API (`reportDetailByPeriod`). Каждая строка = одна финансовая операция с полями: `category`, `subcategory`, `amount`, `date`, `marketplace`, `user_id`.

### SQL RPC `get_costs_tree()`

Агрегирует `mp_costs_details` по `category`, строит дерево:
- `total_accrued = SUM(ALL amounts)` — итоговая выплата от WB
- `percent_base_sales = ABS(SUM(amounts WHERE category='Продажи'))` — база для расчёта долей

### Структура дерева

```
Продажи:          +N  (чистые продажи — ПОЛОЖИТЕЛЬНОЕ)
Вознаграждение:   -N  (комиссия WB)
Доставка:         -N  (логистика)
Хранение:         -N
Эквайринг:        -N
Прочие/СПП:       +N  (CREDIT — положительное, не "Продажи")
Возмещения:       +N  (CREDIT)
```

### Инвариант (проверен на реальных данных)

```
total_accrued = sum(все tree items) = Продажи + Credits + Удержания
```

Пример:
```
Продажи   = +7 866
Credits   = +3 243
Удержания = -2 713
total_accrued = 7 866 + 3 243 + (-2 713) = 8 396
```

### Метрики WB Card

| Метрика | Формула | Источник |
|---------|---------|----------|
| Продажи | `salesItem.amount + creditsTotal` | costs-tree |
| Начислено | `total_accrued` | costs-tree RPC |
| Удержания | `SUM(negative tree items)` | фронтенд |
| Прибыль | `total_accrued - purchase*ratio - ad*ratio` | DashboardPage |

### sale_price в WB

```
sale_price = retail_price_withdisc_rub
```

Это цена после СПП (то, что покупатель реально заплатил + компенсация от WB). Не путать с `price` (каталожная цена до скидок).

Таблица `mp_costs_details` — данные из OZON Finance API (`get_finance_transaction_list`). Операции классифицируются через `_classify_ozon_operation()`.

| Аспект | OZON | WB |

## 12. Сверка (Reconciliation)

Для верификации корректности расчётов используются скрипты сверки, сравнивающие данные API с CSV-выгрузками из Личных кабинетов маркетплейсов.

### WB Reconciliation

**Скрипт:** `wb/reconcile_wb.py`

**Методика:**
1. Скачать CSV из ЛК WB (раздел "Финансовые отчёты")
2. Запустить API-запрос costs-tree за тот же период
3. Сравнить суммы по категориям

**Результат:** расхождение не более 0.03 руб. по всем категориям.

### OZON Reconciliation

**Скрипт:** `ozon/reconcile_accruals.py`

**Методика:**
1. Скачать CSV из ЛК OZON (раздел "Финансы -> Детализация начислений")
2. Запустить API-запрос costs-tree за тот же период
3. Сравнить суммы по 6 категориям

**Результат:** расхождение = 0.00 руб. по ВСЕМ категориям (абсолютная точность).

### Регулярность сверки

Рекомендуется проводить сверку:
- При подключении нового аккаунта МП
- При обнаружении аномалий в прибыли
- После обновления логики sync_service
- Не реже 1 раза в квартал

## 13. Waterfall (Каскад прибыли)

Визуализация структуры прибыли в формате каскадной диаграммы (waterfall chart).

### Формула

```
Выручка (Продажи)
  - Удержания МП
  - Себестоимость
  - Реклама
  = Прибыль (маржа %)
```

### Данные

| Строка | Значение | Цвет |
|--------|----------|------|
| Продажи | `revenueForTile` | Зелёный |
| Удержания МП | `-mpDeductionsForTile` | Серый |
| Себестоимость | `-adjustedPurchase` | Янтарный |
| Реклама | `-adCostForTile` | Фиолетовый |
| Прибыль | `netProfitForTile` | Зелёный / Красный |

Маржа: `margin = (profit / revenue) * 100%`

### Особенности реализации

- Div-based бары (НЕ Recharts) — легковесный компонент
- Пропорции рассчитываются относительно выручки
- Нулевые строки расходов скрываются
- При убытке — красный цвет прибыли и маржи

---

## 14. ProfitChart (Тренд прибыли)

Dual AreaChart (Recharts): выручка + прибыль по дням.

### Формула дневной прибыли (оценка)

```
profitMargin  = netProfitForTile / revenueForTile
dailyProfit_i = dailyRevenue_i * profitMargin
```

Это оценка (estimate), потому что costs-tree не предоставляет данные с дневной гранулярностью — только агрегат за период. Средняя маржа за весь период применяется к каждому дню.

### Визуализация

- Зелёная область (фон) — выручка
- Индиго область (передний план) — прибыль
- Визуальный зазор между ними = расходы
- При отрицательной марже — красный цвет

## 15. Рекламные метрики (Ads)

### KPI рекламных кампаний

| Метрика | Формула | Описание |
|---------|---------|----------|
| Расход | `SUM(cost)` | Общие затраты на рекламу |
| Показы | `SUM(impressions)` | Количество показов объявлений |
| Клики | `SUM(clicks)` | Количество кликов |
| CTR | `clicks / impressions * 100%` | Кликабельность |
| CPC | `cost / clicks` | Стоимость клика |
| Заказы (рекл.) | `SUM(orders_count)` | Заказы, атрибутированные рекламе |
| ДРР | `cost / revenue * 100%` | Доля рекламных расходов |

### Дневной ДРР (график)

```
drr_daily = ad_cost_day / revenue_day * 100%
```

Если `revenue_day = 0`, то `drr_daily = 0` (не infinity).
