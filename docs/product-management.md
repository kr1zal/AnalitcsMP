---

## 1. Проблема

Сейчас товары засеяны вручную через SQL-миграцию (5 SKU админа). При регистрации нового пользователя:
- Товары не создаются автоматически из маркетплейсов
- Нет UI для ввода/редактирования себестоимости
- Нет способа связать один и тот же товар на WB и Ozon (разные штрихкоды)
- Себестоимость нужна для расчёта чистой прибыли на дашборде

## 2. Решение

Добавить на страницу "Настройки" секцию управления товарами:
- **Авто-импорт** — при синке товары автоматически подтягиваются из WB/Ozon API
- **Себестоимость** — редактируемые ячейки для ввода закупочной цены
- **Связывание** — замочек для объединения товаров с разных маркетплейсов
- **Сортировка** — drag&drop (desktop) или стрелки (mobile) для порядка отображения


**Элементы строки товара:**
- `⠿` — drag-handle (GripVertical icon), при зажатии можно перетащить строку вверх/вниз
- Название товара — текст, не редактируемый (приходит из маркетплейса)
- `[280]` — input поле себестоимости (число), сохраняется при потере фокуса (blur)

**Колонка связей (между WB и Ozon):**
- `🔒✓` — закрытый замок + галочка = товары связаны (один физический товар)
- `🔓` — открытый замок = товары НЕ связаны, но можно связать (клик)
- Пустое место — на одной из сторон нет товара на этой строке

## 4. Модель данных

### 4.1 Текущая таблица `mp_products`

| Колонка | Тип | Описание |
|---------|-----|----------|
| id | UUID PK | Auto-generated |
| user_id | UUID FK → auth.users | Владелец |
| barcode | VARCHAR(20) | Штрихкод (уникален для user_id) |
| name | VARCHAR(255) | Название товара |
| purchase_price | DECIMAL(10,2) NOT NULL | **Себестоимость (закупочная цена)** |
| wb_nm_id | BIGINT | WB номенклатура ID |
| wb_vendor_code | VARCHAR(50) | WB артикул |
| ozon_product_id | BIGINT | Ozon product ID |
| ozon_offer_id | VARCHAR(50) | Ozon offer ID (= barcode) |
| ozon_sku | TEXT | Ozon FBO SKU (для analytics API) |
| created_at | TIMESTAMPTZ | Создан |
| updated_at | TIMESTAMPTZ | Обновлён |

### 4.2 Новые колонки (миграция 013)

| Колонка | Тип | Default | Описание |
|---------|-----|---------|----------|
| **sort_order** | INTEGER | 0 | Порядок отображения в столбце |
| **product_group_id** | UUID | NULL | ID группы связанных товаров |

**Constraint:** `UNIQUE (user_id, barcode)` — уже существует.

**Индексы:**
- `idx_mp_products_group` — partial index на `product_group_id WHERE NOT NULL`
- `idx_mp_products_sort` — по `(user_id, sort_order)` для быстрой сортировки

### 4.3 Логика связывания

**Автосвязь (один штрихкод):**
Если WB и Ozon используют одинаковый штрихкод → они попадают в одну строку `mp_products` (и `wb_nm_id`, и `ozon_product_id` заполнены). Это "авто-связанные" товары. `product_group_id = NULL` — связь через общий `id`.

**Ручная связь (разные штрихкоды):**
Если товары на WB и Ozon имеют разные штрихкоды → это две РАЗНЫЕ строки в `mp_products`. Пользователь связывает их замочком → обоим проставляется одинаковый `product_group_id` (новый UUID).

```
Пример:
┌────────────┬────────────────┬────────┬──────────┬─────────────────┬──────────────────┐
│ id         │ name           │barcode │wb_nm_id  │ozon_product_id  │product_group_id  │
├────────────┼────────────────┼────────┼──────────┼─────────────────┼──────────────────┤
│ aaa-111    │ Магний + В6    │ 467015 │ 254327396│ 1144779512      │ NULL             │ ← авто-связан (один barcode)
│ bbb-222    │ Рубашка WB     │ 123456 │ 999111   │ NULL            │ ccc-group-111    │ ← связан замочком
│ bbb-333    │ Рубашка Ozon   │ 789012 │ NULL     │ 555222          │ ccc-group-111    │ ← связан замочком (тот же group)
│ ddd-444    │ L-карнитин     │ 467015 │ 254278127│ 1145915272      │ NULL             │ ← авто-связан
│ eee-555    │ Куртка WB      │ 111222 │ 888333   │ NULL            │ NULL             │ ← НЕ связан (только WB)
└────────────┴────────────────┴────────┴──────────┴─────────────────┴──────────────────┘
```

### 4.4 Влияние на расчёт прибыли

**Формула в RPC `get_dashboard_summary`:**
```sql
SUM(mp_products.purchase_price * mp_sales.sales_count)
```

Каждая строка `mp_sales` ссылается на свой `product_id` из `mp_products`. Связанные товары — это ДВЕ строки с одинаковой `purchase_price`. Поэтому расчёт корректен автоматически — не нужно менять RPC.

**Unit-экономика (будущее):**
Для объединённой статистики по связанным товарам нужно будет GROUP BY `product_group_id`. Но это отдельная фича — сейчас каждый товар показывается отдельно.

### 5.2 PUT /products/{id}/purchase-price

**Тело запроса:**
```json
{ "purchase_price": 350.00 }
```

**Логика:**
1. Найти товар по `id` + `user_id` → 404 если нет
2. Валидация: `purchase_price >= 0`
3. Обновить `purchase_price` и `updated_at`
4. Если `product_group_id IS NOT NULL` → обновить `purchase_price` у ВСЕХ товаров с таким же group_id и user_id

**Ответ:**
```json
{
  "status": "success",
  "product_id": "aaa-111",
  "purchase_price": 350.00,
  "linked_updated": 1
}
```

### 5.3 PUT /products/reorder

**Тело запроса:**
```json
{
  "items": [
    { "product_id": "aaa-111", "sort_order": 0 },
    { "product_id": "bbb-222", "sort_order": 1 },
    { "product_id": "ccc-333", "sort_order": 2 }
  ]
}
```

**Логика:** Обновить `sort_order` для каждого product_id WHERE user_id = current_user.

**Ответ:**
```json
{ "status": "success", "updated": 3 }
```

### 5.4 POST /products/link

**Тело запроса:**
```json
{
  "wb_product_id": "bbb-222",
  "ozon_product_id": "bbb-333",
  "purchase_price": 280.00
}
```

**Логика:**
1. Найти оба товара, проверить user_id
2. Валидация: один должен иметь `wb_nm_id`, другой — `ozon_product_id`
3. Если у одного уже есть `product_group_id` → использовать его, иначе генерировать новый UUID
4. Проставить `product_group_id` обоим
5. Установить `purchase_price` (переданный в body) на ОБА товара

**CC conflict flow (на фронтенде):**
- Фронт проверяет: `wb.purchase_price !== ozon.purchase_price`?
- Если совпадают → POST /products/link сразу
- Если не совпадают → shake + red cells → модалка выбора CC → POST /products/link с выбранной ценой

**Ответ:**
```json
{
  "status": "success",
  "group_id": "ccc-group-111",
  "purchase_price": 280.00
}
```

### 5.5 POST /products/unlink/{group_id}

**Логика:**
1. Найти все товары с `product_group_id = group_id` AND `user_id = current_user`
2. Установить `product_group_id = NULL` на всех
3. `purchase_price` остаётся как есть (не сбрасывается)

**Ответ:**
```json
{ "status": "success", "unlinked_count": 2 }
```

## 6. Авто-импорт товаров (sync_products)

### 6.1 Текущее поведение
```
sync_products():
  barcodes = [из mp_products WHERE user_id = X]
  wb_cards = WB API get_cards_by_barcode(barcodes)  ← фильтр по СУЩЕСТВУЮЩИМ

  → только ОБНОВЛЯЕТ wb_nm_id, ozon_product_id на существующих строках
```

### 6.2 Новое поведение (РЕАЛИЗОВАНО 14.02.2026)

```
sync_products():
  existing = {barcode: product} из mp_products
  max_sku = план пользователя (Free=3, Pro=20, Business=∞)
  allowed_mp = план пользователя (Free=["wb"], Pro/Business=["wb","ozon"])
  current_count = len(existing) - (1 если есть WB_ACCOUNT)

  a) ЕСЛИ "wb" в allowed_mp:
    all_wb_cards = WB API get_cards_list()  ← БЕЗ фильтра, ВСЕ карточки
    ДЛЯ КАЖДОЙ card:
      barcode = card.sizes[].skus[]
      ЕСЛИ barcode в existing:
        UPDATE wb_nm_id, wb_vendor_code
      ИНАЧЕ ЕСЛИ current_count < max_sku:
        INSERT mp_products(barcode, name=card.title, purchase_price=0, wb_nm_id=card.nmID)
        current_count += 1
      ИНАЧЕ:
        пропуск (лимит SKU)

  b) ЕСЛИ "ozon" в allowed_mp:
    all_ozon = Ozon API get_product_list()  ← ВСЕ товары

  c) Ozon: UPDATE ozon_product_id, ozon_offer_id на существующих строках (по offer_id = barcode)

  d) Ozon: Получить названия через /v3/product/info/list → INSERT новых товаров (если < max_sku)
     ⚠ ВАЖНО: ответ API = {"items": [...]} (НЕ {"result": {"items": [...]}})
     Парсинг: info.get("items") or info.get("result", {}).get("items", [])

  e) Ozon: Получить FBO SKU через /v3/product/info/list → sources[].sku
     ⚠ ВАЖНО: source может быть "fbo", "fbs", "sds" (seller direct shipment)
     Логика: предпочтительно "fbo", fallback — любой source с sku
     Fallback: если SKU не найден → get_stocks_on_warehouses() (sku + item_code)

  f) CC из API (best-effort) — см. раздел 6.3

  g) Инвалидировать кэши

  h) ВЕРНУТЬ {status, updated, created, skipped_sku_limit}
```

### 6.2.1 Исправленный баг: Ozon SKU mapping (14.02.2026)

**Симптом:** У пользователя 0 продаж, 0 расходов, 0 заказов по Ozon, хотя sync_all логирует "success". Все 5 товаров имеют `ozon_sku = NULL`.

**Корневая причина 1: формат ответа API**

Ozon API `/v3/product/info/list` возвращает:
```json
{"items": [{"id": 1144779512, "sources": [...], ...}]}
```

Код парсил НЕПРАВИЛЬНО:
```python
# БЫЛО (всегда возвращал пустой список):
for item in info.get("result", {}).get("items", []):
```

Правильный парсинг:
```python
# СТАЛО:
info_items = info.get("items") or info.get("result", {}).get("items", [])
for item in info_items:
```

Этот баг присутствовал в **4 местах** sync_service.py + **1 место** ozon_client.py (get_products_by_barcode).

**Корневая причина 2: фильтр по source type**

Ozon sources для товаров продавца могут быть:
- `"fbo"` — Fulfillment by Ozon (хранение на складе Ozon)
- `"fbs"` — Fulfillment by Seller
- `"sds"` — Seller Direct Shipment

Код фильтровал только `"fbo"`:
```python
# БЫЛО:
if src.get("source") == "fbo" and src.get("sku"):
```

Реальные товары использовали `"sds"` → ozon_sku НИКОГДА не заполнялся.

Исправлено: предпочитать `"fbo"`, fallback на любой source:
```python
# СТАЛО:
for preferred_source in ("fbo", None):
    for src in item.get("sources", []):
        if preferred_source and src.get("source") != preferred_source:
            continue
        if src.get("sku"):
            ozon_sku = str(src["sku"])
            break
    if ozon_sku:
        break
```

**Дополнительный fallback:** Если после всех попыток ozon_sku не найден — запрашиваем `get_stocks_on_warehouses()` и маппим по `item_code` (= offer_id = barcode).

**Влияние:** Без ozon_sku → `ozon_sku_map` пуст → ВСЕ Ozon данные (sales, costs, orders) тихо пропускались с warning "не найден в маппинге".

**Файлы исправлений:**
- `backend/app/services/sync_service.py` — 6 правок (4× парсинг ответа, 1× source filter, 1× warehouse fallback)
- `backend/app/services/ozon_client.py` — 1 правка (парсинг в get_products_by_barcode)

### 6.3 CC из API маркетплейсов (best-effort, ПРОВЕРЕНО 14.02.2026)

При синке пробуем получить себестоимость из API:

- **WB:** API карточек (`get_cards_list`) НЕ отдаёт себестоимость. Поля: `nmID`, `vendorCode`, `title`, `sizes`, `photos` — нет cost_price.

- **Ozon:** **ПОДТВЕРЖДЕНО — API НЕ отдаёт себестоимость.**
  - `/v5/product/info/prices` возвращает: `price`, `old_price`, `marketing_seller_price`, `min_price`, `retail_price`, `vat`, `net_price` — **нет `cost_price`**
  - `/v3/product/info/list` возвращает: `price`, `old_price`, `min_price` — **нет `cost_price`**
  - В личном кабинете Ozon Seller Dashboard "себестоимость" отображается, но через API не экспонируется
  - Попытки эндпоинтов `/v1`, `/v2`, `/v4/product/info/prices` → 404. Рабочий: `/v5/product/info/prices`

- **Вывод:** Пользователи ДОЛЖНЫ вводить `purchase_price` вручную в Настройках (ProductManagement)

- **Код:** Step (f) в sync_products() пытается найти `cost_price` / `purchase_price` в ответе Ozon prices API. Если не найдено (а оно не найдено) — gracefully пропускает. НИКОГДА не перезаписывает ручной ввод (только если `purchase_price = 0`).

## 9. Edge cases и ограничения

### 9.1 Авто-связанные товары (один barcode)

Товары, у которых совпал barcode на WB и Ozon — это ОДНА строка в БД. Они показываются в обоих столбцах, но замочек между ними НЕ кликабельный (постоянно закрыт, серый). Их нельзя "развязать" — они связаны на уровне barcode.

### 9.2 SKU лимит превышен после даунгрейда

Пользователь с Pro (20 SKU, 15 товаров) переходит на Free (3 SKU):
- Все 15 товаров остаются
- Новые товары не импортируются при sync
- UI показывает "SKU 15/3" красным + текст "Лимит превышен. Для импорта новых товаров перейдите на Pro."
- Редактирование существующих товаров работает

### 9.3 Ozon на Free плане

Free план: `marketplaces: ["wb"]`. Ozon колонка показывается заблюренной с оверлеем "Доступно на Pro" + кнопка апгрейда.

### 9.4 Нет товаров (пустое состояние)

```
┌─ Товары ──────────────────── SKU 0/20 ─┐
│                                          │
│  Товары появятся после первой            │
│  синхронизации с маркетплейсами.        │
│                                          │
│  Добавьте API-токены ниже и нажмите     │
│  "Сохранить и обновить".                │
│                                          │
└──────────────────────────────────────────┘
```

### 9.5 WB_ACCOUNT системный товар

Исключается из UI и подсчёта SKU. Фильтр: `barcode !== "WB_ACCOUNT"`.

### 9.6 Длинные названия товаров

Обрезаются с `text-ellipsis overflow-hidden whitespace-nowrap`. Полное название — в title/tooltip при наведении.
