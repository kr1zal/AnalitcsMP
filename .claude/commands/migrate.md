Безопасное применение миграции БД в Supabase.

Аргумент $ARGUMENTS — номер миграции или путь к SQL файлу.

## Шаг 1: Найди миграцию

Если указан номер (например "014"):
```bash
ls backend/migrations/*014*
```

Если указан путь — используй его. Если ничего не указано — покажи список:
```bash
ls -la backend/migrations/*.sql
```

## Шаг 2: Прочитай и проанализируй SQL

Прочитай файл миграции и проверь:
- ❗ **DROP TABLE / DROP COLUMN** — СТОП. Предупреди пользователя о потере данных.
- ❗ **ALTER TABLE ... DROP** — СТОП. Может сломать существующие данные.
- ❗ **TRUNCATE** — СТОП. Удаление всех данных.
- ⚠️ **ALTER TABLE ... ADD COLUMN NOT NULL** без DEFAULT — может упасть если есть данные.
- ⚠️ **CREATE POLICY** — проверь что не ломает существующие политики RLS.
- ✅ **CREATE TABLE, ADD COLUMN, CREATE INDEX** — безопасно.

## Шаг 5: Верификация (после подтверждения пользователя)

Если есть MCP Supabase — проверь через SQL:
```sql
SELECT column_name, data_type FROM information_schema.columns WHERE table_name = 'TABLE_NAME';
```

Если нет MCP — проверь через API endpoint который использует новую таблицу/колонку.

ВАЖНО: НИКОГДА не применяй миграции автоматически. Только показывай SQL и жди подтверждения.
